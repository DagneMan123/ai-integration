DETECTION FLOW:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Scenario 1: Health Check Detects Disconnection
─────────────────────────────────────────────────
  Every 30 seconds:
    ┌─ Health Check Runs
    │  └─ SELECT 1 query fails
    │     └─ Connection error detected
    │        └─ handleDisconnection() triggered
    │           └─ Automatic reconnection begins
    │              └─ System recovers
    └─ Logs: "🚨 Health check detected database disconnection!"

Scenario 2: Prisma Error Event Detected
─────────────────────────────────────────
  During normal operation:
    ┌─ Prisma query fails
    │  └─ Error event emitted
    │     └─ Error listener catches it
    │        └─ Checks if connection-related
    │           └─ handleDisconnection() triggered
    │              └─ Automatic reconnection begins
    │                 └─ System recovers
    └─ Logs: "❌ Prisma Error: Can't reach database server"

Scenario 3: Request Arrives During Disconnection
──────────────────────────────────────────────────
  User makes API request:
    ┌─ Request hits connection check middleware
    │  └─ Health check runs
    │     └─ Connection fails
    │        └─ Returns 503 error
    │           └─ User sees clear error message
    │              └─ No silent failure
    └─ Logs: "Database connection check failed"


ERROR DETECTION PATTERNS:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

System detects these connection errors:
  ✓ "Can't reach database server"
  ✓ "connection refused"
  ✓ "ECONNREFUSED"
  ✓ "ETIMEDOUT"
  ✓ "socket hang up"

When detected:
  1. Error is logged with full details
  2. handleDisconnection() is triggered immediately
  3. Automatic reconnection begins
  4. System attempts up to 5 reconnection tries
  5. If successful, system resumes normal operation
  6. If failed, retries again in 10 seconds


MIDDLEWARE ROUTING:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

STRICT CHECK (connectionCheck):
  ├─ /api/payments/*        ← Payment operations (CRITICAL)
  └─ /api/wallet/*          ← Wallet operations (CRITICAL)
     └─ Blocks if database unavailable
     └─ Returns 503 error
     └─ Prevents silent failures

LIGHT CHECK (lightConnectionCheck):
  ├─ /api/auth/*            ← Authentication
  ├─ /api/users/*           ← User operations
  ├─ /api/interviews/*      ← Interview operations
  └─ /api/applications/*    ← Application operations
     └─ 2-second timeout
     └─ Doesn't block if slow
     └─ Logs warnings
     └─ Allows graceful degradation


RECOVERY TIMELINE:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

T+0s:   Database disconnects
        └─ Prisma error event fires
           └─ handleDisconnection() triggered

T+0-1s: Attempt 1 to reconnect
        └─ Fails (database still down)
           └─ Wait 1 second

T+1-3s: Attempt 2 to reconnect
        └─ Fails (database still down)
           └─ Wait 2 seconds

T+3-7s: Attempt 3 to reconnect
        └─ Fails (database still down)
           └─ Wait 4 seconds

T+7-15s: Attempt 4 to reconnect
         └─ Fails (database still down)
            └─ Wait 8 seconds

T+15-31s: Attempt 5 to reconnect
          └─ Fails (database still down)
             └─ Schedule retry in 10 seconds

T+31-41s: Retry attempt
          └─ Succeeds (database back online)
             └─ System fully recovered

Total recovery time: ~40 seconds (worst case)
