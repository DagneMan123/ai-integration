# SimuAI Automated Billing & Wallet System - Requirements Document

## Introduction

The SimuAI Automated Billing & Wallet System is a credit-based financial engine that enables Candidates to purchase "AI Interview Credits" through the Chapa payment gateway (supporting Telebirr, CBEBirr, and Banks). The system automates transaction processing, wallet management, and provides real-time financial visibility without manual intervention. The system enforces a 1 Credit = 5 ETB conversion rule and ensures atomic, idempotent payment processing with comprehensive audit logging.

## Glossary

- **Candidate**: A user who purchases and consumes AI Interview Credits
- **AI Interview Credit**: A unit of currency representing the right to conduct one AI-powered interview session
- **Wallet**: A user account holding the current balance of AI Interview Credits
- **Payment**: A financial transaction initiated by a Candidate to purchase credits
- **Transaction Reference (tx_ref)**: A unique identifier generated by the system for each payment attempt
- **Chapa**: Third-party payment gateway supporting multiple payment methods
- **Webhook**: Callback endpoint that receives payment status updates from Chapa
- **Server-Side Verification**: Process of validating payment completion using Chapa's secret key
- **Idempotent**: Property ensuring that processing the same transaction multiple times produces the same result
- **Credit Bundle**: Predefined package of credits at a fixed price (e.g., 5 Credits for 25 ETB)
- **Payment Status**: State of a transaction (PENDING, COMPLETED, FAILED)
- **Billing & History Table**: User-facing record of all payment transactions
- **Financial Analytics**: Aggregated metrics calculated from payment and wallet data
- **Atomic Transaction**: Database operation that either completes fully or not at all, preventing partial updates
- **Optimistic Locking**: Concurrency control mechanism preventing race conditions in wallet updates

## Requirements

### Requirement 1: Credit Bundle Configuration and Display

**User Story:** As a Candidate, I want to see predefined credit bundles with clear pricing, so that I can quickly understand my purchase options.

#### Acceptance Criteria

1. THE System SHALL display at least three predefined credit bundles on the dashboard
2. EACH credit bundle SHALL display the number of credits and the price in ETB
3. THE System SHALL enforce the 1 Credit = 5 ETB conversion rule for all bundles (e.g., 5 Credits = 25 ETB, 10 Credits = 50 ETB, 20 Credits = 100 ETB)
4. WHERE a credit bundle is configurable, THE System SHALL allow administrators to modify bundle amounts and prices while maintaining the 5 ETB per credit ratio
5. THE System SHALL validate that all bundle prices are positive numbers before displaying them
6. THE System SHALL display credit bundles prominently on the Candidate dashboard with a "Top Up" button adjacent to each bundle

### Requirement 2: Payment Initialization

**User Story:** As a Candidate, I want to initiate a payment for credits, so that I can purchase AI Interview Credits through Chapa.

#### Acceptance Criteria

1. WHEN a Candidate clicks the "Top Up" button for a credit bundle, THE System SHALL generate a unique transaction reference (tx_ref)
2. THE System SHALL validate that the selected credit bundle exists in the database
3. THE System SHALL validate that the bundle amount is a positive number
4. THE System SHALL create a Payment record with status PENDING before redirecting to Chapa
5. THE System SHALL pass metadata to Chapa containing the userId and credit_amount (mandatory for automation)
6. THE System SHALL respond with a payment URL and transaction reference within 2 seconds
7. IF the credit bundle does not exist, THEN THE System SHALL return a 404 error with a descriptive message
8. IF the bundle amount is not positive, THEN THE System SHALL return a 400 error with validation details
9. THE Endpoint POST /api/payments/initialize SHALL accept requests containing bundleId and userId

### Requirement 3: Webhook Reception and Idempotency

**User Story:** As the System, I want to receive payment status updates from Chapa, so that I can process successful payments automatically.

#### Acceptance Criteria

1. THE System SHALL provide a Webhook endpoint to receive payment status callbacks from Chapa
2. WHEN Chapa sends a webhook notification, THE System SHALL extract the transaction reference (tx_ref) from the payload
3. THE System SHALL verify that the same tx_ref is not processed more than once (idempotent processing)
4. IF the same tx_ref is received multiple times, THEN THE System SHALL return success without duplicating credits
5. THE System SHALL handle webhook timeouts gracefully by implementing retry logic
6. THE System SHALL log all webhook events with timestamp and status for audit purposes
7. WHEN a webhook is received, THE System SHALL respond with HTTP 200 within 5 seconds to acknowledge receipt

### Requirement 4: Server-Side Payment Verification

**User Story:** As the System, I want to verify payment completion with Chapa before granting credits, so that I prevent fraudulent or incomplete transactions.

#### Acceptance Criteria

1. WHEN a webhook is received, THE System SHALL perform server-side verification using Chapa's secret key
2. THE System SHALL validate Chapa's signature on the webhook payload
3. IF the signature is invalid, THEN THE System SHALL reject the webhook and log a security event
4. IF the payment status from Chapa is COMPLETED, THEN THE System SHALL proceed to credit the wallet
5. IF the payment status from Chapa is FAILED, THEN THE System SHALL update the Payment record status to FAILED and not modify the wallet
6. THE System SHALL retrieve the original Payment record using the tx_ref
7. THE System SHALL verify that the payment amount matches the original request

### Requirement 5: Atomic Wallet Credit Update

**User Story:** As the System, I want to update the wallet atomically when payment succeeds, so that credits are granted reliably without duplication or loss.

#### Acceptance Criteria

1. WHEN payment verification succeeds, THE System SHALL execute a Prisma transaction containing three operations
2. THE System SHALL create a Payment record with status COMPLETED, amount, creditAmount, and paidAt timestamp
3. THE System SHALL increment the Wallet balance by the creditAmount for the specific userId
4. THE System SHALL log the transaction with userId, amount, creditAmount, status, and timestamp
5. IF any operation in the transaction fails, THEN THE System SHALL rollback all changes and not modify the wallet
6. THE System SHALL use optimistic locking to prevent concurrent wallet updates from causing race conditions
7. THE System SHALL ensure that a single tx_ref cannot be processed more than once to prevent credit duplication

### Requirement 6: Real-Time Wallet Balance Display

**User Story:** As a Candidate, I want to see my current AI Credit balance on the dashboard, so that I know how many interviews I can conduct.

#### Acceptance Criteria

1. THE System SHALL display the current AI Credit Balance on the Candidate dashboard
2. WHEN a payment is successfully processed, THE System SHALL update the wallet balance display within 5 seconds
3. THE System SHALL display the balance on all Candidate pages for quick reference
4. THE System SHALL retrieve the wallet balance from the database for each page load
5. THE System SHALL display the balance with the currency label "Credits"
6. IF the wallet does not exist for a Candidate, THEN THE System SHALL initialize it with a balance of 0

### Requirement 7: Transaction History Population

**User Story:** As a Candidate, I want to see a complete history of my payments, so that I can track my spending and verify transactions.

#### Acceptance Criteria

1. WHEN a payment is successfully completed, THE System SHALL automatically populate the Billing & History table with transaction details
2. THE System SHALL record the Date & Time of the transaction in ISO 8601 format
3. THE System SHALL record the Transaction Reference (tx_ref) for each payment
4. THE System SHALL record the Amount Paid in ETB
5. THE System SHALL record the Payment Status (SUCCESS or FAILED)
6. THE System SHALL record the Payment Method (telebirr, cbebirr, or bank)
7. THE System SHALL record the Chapa Reference (Chapa's unique ID) for reconciliation
8. THE System SHALL display transactions in reverse chronological order (newest first)
9. THE System SHALL include all payment records regardless of status (COMPLETED or FAILED)

### Requirement 8: Transaction History Sorting and Filtering

**User Story:** As a Candidate, I want to sort and filter my transaction history, so that I can find specific payments quickly.

#### Acceptance Criteria

1. THE System SHALL allow sorting transaction history by Date, Amount, and Status
2. THE System SHALL allow filtering by Payment Status (SUCCESS, FAILED, PENDING)
3. THE System SHALL allow filtering by Payment Method (telebirr, cbebirr, bank)
4. THE System SHALL allow filtering by date range (from date to date)
5. THE System SHALL return filtered results within 1 second
6. THE System SHALL display the total number of transactions matching the filter criteria
7. THE System SHALL support pagination for transaction history with configurable page size

### Requirement 9: Transaction History Export

**User Story:** As a Candidate, I want to export my transaction history, so that I can maintain records for personal accounting.

#### Acceptance Criteria

1. THE System SHALL provide an "Export" button on the Billing & History page
2. WHEN the Export button is clicked, THE System SHALL generate a CSV file containing all transaction records
3. THE System SHALL include columns for Date, Transaction Reference, Amount, Status, and Payment Method
4. THE System SHALL apply current filters and sorting to the exported data
5. THE System SHALL name the exported file with the format "billing_history_[date].csv"
6. THE System SHALL trigger a browser download of the CSV file

### Requirement 10: Financial Analytics Dashboard Cards

**User Story:** As a Candidate, I want to see aggregated financial metrics, so that I can understand my spending patterns.

#### Acceptance Criteria

1. THE System SHALL display a "Total Spent" card showing the sum of all successful payments in ETB
2. THE System SHALL display a "Successful Transactions" card showing the count of completed payments
3. THE System SHALL display an "Average Transaction Value" card showing the mean amount per successful transaction
4. THE System SHALL display a "Credits Remaining" card showing the current wallet balance
5. THE System SHALL calculate all metrics dynamically from the database on each page load
6. THE System SHALL display metrics with appropriate currency labels (ETB for money, Credits for balance)
7. THE System SHALL display metrics with two decimal places for currency values
8. IF no transactions exist, THEN THE System SHALL display 0 for all metrics

### Requirement 11: Interview Start Access Control

**User Story:** As the System, I want to prevent interviews from starting without sufficient credits, so that only paid users can conduct interviews.

#### Acceptance Criteria

1. WHEN a Candidate attempts to start an AI Interview, THE System SHALL check the wallet balance
2. IF the wallet balance is less than 1 credit, THEN THE System SHALL disable the "Start AI Interview" button
3. IF the wallet balance is less than 1 credit, THEN THE System SHALL display a clear "Top Up" prompt with a link to the payment interface
4. IF the wallet balance is 1 or more credits, THEN THE System SHALL enable the "Start AI Interview" button
5. THE System SHALL display the current credit balance next to the "Start AI Interview" button
6. THE System SHALL prevent interview creation through the API if the wallet balance is insufficient

### Requirement 12: Atomic Credit Deduction on Interview Start

**User Story:** As the System, I want to deduct credits atomically when an interview starts, so that credits are consumed reliably.

#### Acceptance Criteria

1. WHEN an interview starts successfully, THE System SHALL deduct exactly 1 credit from the wallet
2. THE System SHALL execute the credit deduction as an atomic database transaction
3. IF the interview start fails, THEN THE System SHALL not deduct any credits
4. THE System SHALL log the credit deduction with the interviewId, userId, and timestamp
5. THE System SHALL verify the wallet has sufficient balance before deducting credits
6. IF the wallet balance is insufficient, THEN THE System SHALL return a 402 error (Payment Required)

### Requirement 13: Interview Cancellation and Credit Refund

**User Story:** As a Candidate, I want to receive a credit refund if I cancel an interview, so that I don't lose credits for cancelled sessions.

#### Acceptance Criteria

1. WHEN a Candidate cancels an interview, THE System SHALL refund exactly 1 credit to the wallet
2. THE System SHALL execute the credit refund as an atomic database transaction
3. THE System SHALL update the interview status to CANCELLED
4. THE System SHALL log the refund with the interviewId, userId, and timestamp
5. IF the interview was already completed, THEN THE System SHALL not refund credits
6. IF the interview was already cancelled, THEN THE System SHALL not refund credits again (idempotent)

### Requirement 14: Payment Data Integrity and Audit Logging

**User Story:** As the System, I want to maintain immutable audit logs of all financial transactions, so that I can comply with audit requirements.

#### Acceptance Criteria

1. THE System SHALL log all payment transactions with userId, amount, creditAmount, status, and timestamp
2. THE System SHALL ensure that payment logs are immutable (cannot be modified after creation)
3. THE System SHALL log all webhook events including timestamp, tx_ref, status, and verification result
4. THE System SHALL log all credit deductions with interviewId, userId, and timestamp
5. THE System SHALL log all credit refunds with interviewId, userId, and timestamp
6. THE System SHALL log all failed payment verification attempts with reason and timestamp
7. THE System SHALL retain all audit logs for a minimum of 7 years for compliance
8. THE System SHALL provide an audit log export endpoint for administrators

### Requirement 15: Concurrent Payment Request Handling

**User Story:** As the System, I want to handle concurrent payment requests safely, so that multiple simultaneous payments don't cause data corruption.

#### Acceptance Criteria

1. THE System SHALL use optimistic locking on the Wallet table to prevent race conditions
2. WHEN multiple payments for the same user are processed concurrently, THE System SHALL ensure each credit update is applied correctly
3. THE System SHALL implement database-level constraints to prevent duplicate tx_ref processing
4. THE System SHALL use Prisma transactions to ensure atomicity of multi-step operations
5. IF a concurrent update conflict occurs, THEN THE System SHALL retry the operation up to 3 times
6. IF retries are exhausted, THEN THE System SHALL log the conflict and return a 409 error

### Requirement 16: Failed Payment Handling

**User Story:** As the System, I want to handle failed payments gracefully, so that users understand why their payment didn't succeed.

#### Acceptance Criteria

1. WHEN a payment fails at Chapa, THE System SHALL update the Payment record status to FAILED
2. THE System SHALL not modify the wallet balance for failed payments
3. THE System SHALL log the failure reason from Chapa
4. THE System SHALL display a clear error message to the Candidate explaining the failure
5. THE System SHALL provide a "Retry Payment" option for failed transactions
6. THE System SHALL allow the Candidate to select a different payment method for retry

### Requirement 17: Chapa API Downtime Resilience

**User Story:** As the System, I want to handle Chapa API downtime gracefully, so that temporary outages don't break the payment flow.

#### Acceptance Criteria

1. IF Chapa API is unavailable during payment initialization, THEN THE System SHALL return a 503 error with a user-friendly message
2. THE System SHALL implement exponential backoff retry logic for failed Chapa API calls
3. THE System SHALL retry failed webhook verifications up to 5 times with increasing delays
4. THE System SHALL log all Chapa API errors with timestamp and error details
5. THE System SHALL display a maintenance message to users if Chapa is unavailable
6. THE System SHALL queue failed webhook events for later processing

### Requirement 18: Payment Endpoint Authentication

**User Story:** As the System, I want to secure payment endpoints, so that only authenticated users can initiate payments.

#### Acceptance Criteria

1. THE System SHALL require authentication for the POST /api/payments/initialize endpoint
2. THE System SHALL validate the user's JWT token before processing payment requests
3. IF the request lacks a valid JWT token, THEN THE System SHALL return a 401 error
4. THE System SHALL verify that the userId in the request matches the authenticated user
5. IF the userId does not match, THEN THE System SHALL return a 403 error
6. THE System SHALL log all payment requests with the authenticated userId

### Requirement 19: Webhook Signature Validation

**User Story:** As the System, I want to validate webhook signatures, so that I only process legitimate Chapa notifications.

#### Acceptance Criteria

1. THE System SHALL retrieve the Chapa secret key from environment variables
2. WHEN a webhook is received, THE System SHALL compute the expected signature using the secret key
3. THE System SHALL compare the computed signature with the signature in the webhook header
4. IF the signatures do not match, THEN THE System SHALL reject the webhook and log a security event
5. THE System SHALL not process any payment updates from unsigned or invalid webhooks
6. THE System SHALL return a 401 error for invalid signatures

### Requirement 20: Secure Environment Variable Management

**User Story:** As the System, I want to protect sensitive credentials, so that payment security is maintained.

#### Acceptance Criteria

1. THE System SHALL store the Chapa secret key in environment variables (not in code)
2. THE System SHALL store the Chapa API key in environment variables (not in code)
3. THE System SHALL never log sensitive credentials
4. THE System SHALL use HTTPS for all payment-related API calls
5. THE System SHALL validate that required environment variables are set at startup
6. IF required environment variables are missing, THEN THE System SHALL fail startup with a clear error message

### Requirement 21: Payment Initialization Response Time

**User Story:** As a Candidate, I want payment initialization to be fast, so that I don't experience delays when purchasing credits.

#### Acceptance Criteria

1. THE System SHALL respond to payment initialization requests within 2 seconds
2. THE System SHALL create the Payment record and generate the tx_ref within 1 second
3. THE System SHALL retrieve the Chapa payment URL within 1 second
4. IF response time exceeds 2 seconds, THEN THE System SHALL log a performance warning

### Requirement 22: Wallet Balance Update Latency

**User Story:** As a Candidate, I want my wallet balance to update quickly after payment, so that I can start interviews immediately.

#### Acceptance Criteria

1. WHEN a payment is verified and processed, THE System SHALL update the wallet balance within 5 seconds
2. THE System SHALL update the dashboard display within 5 seconds of wallet update
3. THE System SHALL use real-time updates (WebSocket or polling) to reflect balance changes
4. IF balance update exceeds 5 seconds, THEN THE System SHALL log a performance warning

### Requirement 23: Transaction History Query Performance

**User Story:** As a Candidate, I want transaction history to load quickly, so that I can review my payments without delays.

#### Acceptance Criteria

1. THE System SHALL return transaction history queries within 1 second
2. THE System SHALL index the Payment table on userId, createdAt, and status for query optimization
3. THE System SHALL implement pagination to limit the number of records returned per query
4. THE System SHALL cache frequently accessed transaction history for 5 minutes
5. IF query time exceeds 1 second, THEN THE System SHALL log a performance warning

### Requirement 24: Concurrent Payment Processing Scalability

**User Story:** As the System, I want to handle multiple concurrent payments, so that the system scales with user demand.

#### Acceptance Criteria

1. THE System SHALL support at least 100 concurrent payment requests
2. THE System SHALL use connection pooling for database connections
3. THE System SHALL implement rate limiting to prevent abuse (max 10 payment requests per user per minute)
4. THE System SHALL queue excess requests and process them in order
5. IF rate limit is exceeded, THEN THE System SHALL return a 429 error

### Requirement 25: Payment Data Encryption in Transit

**User Story:** As the System, I want to encrypt payment data in transit, so that sensitive information is protected.

#### Acceptance Criteria

1. THE System SHALL use HTTPS (TLS 1.2 or higher) for all payment-related endpoints
2. THE System SHALL enforce HTTPS redirects for all payment URLs
3. THE System SHALL validate SSL certificates for all external API calls to Chapa
4. THE System SHALL not transmit sensitive data (credit card numbers, secrets) in logs or error messages
5. THE System SHALL use secure headers (HSTS, X-Content-Type-Options, X-Frame-Options)

### Requirement 26: Wallet Initialization for New Users

**User Story:** As the System, I want to initialize wallets for new users, so that they can immediately purchase credits.

#### Acceptance Criteria

1. WHEN a new Candidate user is created, THE System SHALL automatically create a Wallet record
2. THE System SHALL set the initial balance to 0 credits
3. THE System SHALL set the currency to "ETB"
4. THE System SHALL record the createdAt and updatedAt timestamps
5. IF wallet creation fails, THEN THE System SHALL log the error and prevent user registration

### Requirement 27: Payment Method Tracking

**User Story:** As an Administrator, I want to track which payment methods users prefer, so that I can optimize payment options.

#### Acceptance Criteria

1. THE System SHALL record the payment method (telebirr, cbebirr, or bank) for each transaction
2. THE System SHALL allow filtering transaction history by payment method
3. THE System SHALL calculate the count and total amount for each payment method
4. THE System SHALL display payment method statistics on the admin analytics dashboard

### Requirement 28: Chapa Reference Reconciliation

**User Story:** As an Administrator, I want to reconcile payments with Chapa, so that I can verify all transactions are accounted for.

#### Acceptance Criteria

1. THE System SHALL store the Chapa Reference (Chapa's unique ID) for each payment
2. THE System SHALL allow searching payments by Chapa Reference
3. THE System SHALL provide a reconciliation report comparing system payments with Chapa records
4. THE System SHALL flag discrepancies between system and Chapa records for investigation

### Requirement 29: Payment Metadata Storage

**User Story:** As the System, I want to store additional payment metadata, so that I can support future features and debugging.

#### Acceptance Criteria

1. THE System SHALL store metadata as JSON in the Payment record
2. THE System SHALL include userId and credit_amount in metadata (mandatory)
3. THE System SHALL include additional fields such as user email, bundle name, and device information
4. THE System SHALL allow querying payments by metadata fields
5. THE System SHALL validate metadata structure before storing

### Requirement 30: Pretty Printer for Payment Records

**User Story:** As a Developer, I want to format payment records for display, so that transaction data is human-readable.

#### Acceptance Criteria

1. THE System SHALL provide a Pretty_Printer function that formats Payment records
2. THE Pretty_Printer SHALL display all payment fields in a readable format
3. THE Pretty_Printer SHALL format dates in a user-friendly format (e.g., "Jan 15, 2024 at 2:30 PM")
4. THE Pretty_Printer SHALL format amounts with currency symbols and two decimal places
5. THE Pretty_Printer SHALL be used for transaction history display and export

### Requirement 31: Payment Record Round-Trip Property

**User Story:** As a Developer, I want to verify payment serialization, so that payment data integrity is maintained.

#### Acceptance Criteria

1. FOR ALL valid Payment records, serializing then deserializing SHALL produce an equivalent object (round-trip property)
2. THE System SHALL support serialization to JSON format
3. THE System SHALL support deserialization from JSON format
4. THE System SHALL preserve all payment fields during round-trip serialization
5. THE System SHALL validate that deserialized records match the original records exactly

### Requirement 32: Wallet Balance Invariant

**User Story:** As the System, I want to maintain wallet balance invariants, so that balances remain consistent.

#### Acceptance Criteria

1. THE System SHALL ensure that wallet balance is always greater than or equal to 0
2. THE System SHALL ensure that wallet balance is always an integer (whole credits only)
3. THE System SHALL ensure that wallet balance changes are always logged
4. THE System SHALL prevent negative balance updates through database constraints
5. THE System SHALL validate balance invariants on every wallet update

### Requirement 33: Payment Status Idempotence

**User Story:** As the System, I want to ensure payment processing is idempotent, so that duplicate webhooks don't cause issues.

#### Acceptance Criteria

1. WHEN the same tx_ref is processed multiple times, THE System SHALL produce the same result each time
2. THE System SHALL not create duplicate Payment records for the same tx_ref
3. THE System SHALL not credit the wallet multiple times for the same tx_ref
4. THE System SHALL return success for duplicate webhook processing
5. THE System SHALL log duplicate webhook attempts for monitoring

### Requirement 34: Payment Status Transitions

**User Story:** As the System, I want to enforce valid payment status transitions, so that payment state remains consistent.

#### Acceptance Criteria

1. THE System SHALL allow transitions from PENDING to COMPLETED
2. THE System SHALL allow transitions from PENDING to FAILED
3. THE System SHALL not allow transitions from COMPLETED to any other status
4. THE System SHALL not allow transitions from FAILED to any other status
5. THE System SHALL log all status transitions with timestamp and reason

### Requirement 35: Wallet Update Conflict Resolution

**User Story:** As the System, I want to resolve concurrent wallet update conflicts, so that all updates are applied correctly.

#### Acceptance Criteria

1. WHEN concurrent wallet updates occur, THE System SHALL use optimistic locking to detect conflicts
2. THE System SHALL retry conflicting updates up to 3 times
3. THE System SHALL apply updates in the order they are received
4. THE System SHALL log all conflict resolutions with details
5. IF conflicts cannot be resolved after retries, THEN THE System SHALL return a 409 error

## Acceptance Criteria Summary

This requirements document defines 35 requirements covering:

- **User Interface**: Credit bundle display, wallet balance visibility, transaction history
- **Payment Processing**: Initialization, verification, webhook handling, idempotency
- **Financial Operations**: Credit deduction, refunds, balance updates, analytics
- **Data Integrity**: Atomic transactions, audit logging, conflict resolution
- **Security**: Authentication, signature validation, encryption, credential management
- **Performance**: Response times, query optimization, concurrent processing
- **Reliability**: Error handling, retry logic, graceful degradation
- **Compliance**: Audit logging, immutable records, reconciliation

All requirements follow EARS patterns and comply with INCOSE quality rules for clarity, testability, completeness, and positive statements.
